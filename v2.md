# Study 2 — Critical Review & V2 Proposal

An honest evaluation of the study design in `README.md` and the `eval/` implementation. Organized as: strengths worth keeping, critical logical flaws that need to be fixed before the study runs, weaknesses that are addressable, gaps to fill, and the V2 proposal.

---

## Strengths

**1. The "thought amplifier" framing is original.**
Most NLP evaluation studies test summarization, factual QA, or instruction following. This one tests a task that doesn't have an established benchmark: *does the model preserve the user's intellectual substance while reformatting it?* That's a real contribution. The distinction between "this is not a summarization study" and what it actually is — a study of fidelity under transformation — is sharp and well-articulated.

**2. Operationalizing voice from a qualitative spec.**
Taking the taya-voice skill and converting it to a scored rubric is methodologically interesting. The decision to avoid LLM-as-judge and use regex+spaCy is correct: it's reproducible, transparent, and falsifiable. Even where the rubric is flawed (and it is — see below), the approach is sound.

**3. Mode-aware evaluation.**
Testing blog and LinkedIn separately with different rubrics reflects a real difference in expected features. Conflating them would add noise. This is correct.

**4. The 2D tradeoff visualization.**
Plotting substance × voice per condition is the right analytical frame. The quadrant interpretation is immediately readable.

**5. Cross-study design with Study 1.**
Sharing inputs enables comparison between the engineering question (Study 1) and the research question (Study 2). Good experimental design.

## Critical Logical Flaws

These are not minor issues. Each one could invalidate a core finding if left unaddressed.

### Flaw 1: Substance fidelity measures topic overlap, not substance preservation

This is the most serious problem in the study.

The metric computes cosine similarity between sentence embeddings of user notes and model output. High similarity means the output contains a *semantically similar sentence*. It does **not** mean the output preserved the user's *argument, framing, or intellectual substance*.

Example:

- **User note:** "The MSE model scores best but produces gray outputs because averaging minimizes squared error — the visual failure mode is the math made visible."
- **Model output:** "MSE models tend to produce lower quality visual outputs despite achieving good scores."

These embed to high cosine similarity — same topic, same basic claim. But the substance is gone. The specific insight (the failure mode IS the math, not a side effect of it) has been flattened into a generic statement. The model replaced the thought with a summary of the thought.

**This is the exact failure mode the study claims to detect, and the metric cannot detect it.** The study would report high substance fidelity for outputs that have lost the substance.

**Fix:** Decompose notes into argument components — claim, evidence, connecting logic, implication — and check each independently. If a note contains "X because Y — which means Z," the metric should verify that X, Y, and Z all appear, and that the causal connective ("because") or the implication ("which means") is preserved, not just the topic.

### Flaw 2: The voice rubric measures surface markers, not voice

The rubric checks for:
- Presence of "More specifically"
- Em-dash usage
- No bullet points
- Contractions present
- Hedging words present

These are **stylistic markers**. An LLM prompted with "include 'more specifically' at least once and use an em-dash" would pass this rubric trivially without writing in Taya's voice. The rubric tests what would be easy for a model to mimic and misses what would be hard.

The taya-voice skill defines deeper structural patterns that the rubric doesn't capture:

| Voice feature (from skill) | Rubric captures it? |
|---|---|
| "Thinks in examples first, generalizes second" | No |
| Subordinate clause density ("the sentence reaches forward") | No |
| Claim → evidence → implication paragraph structure | Rough proxy only (`_paragraphs_end_on_consequence`) |
| "Concession-and-redirect" counterarguments | No |
| "Build to the thesis, don't open with it" | Very narrow check (thesis-in-sentence-1 only) |
| No dictionary definitions as openers | No |
| Mid-paragraph layering (circles back to add nuance) | No |
| "More specifically" as zoom-in move | Yes — but only checks presence, not function |

A model that scatters em-dashes and "more specifically" through a listicle would score well. A model that actually builds arguments the way Taya does — examples before generalizations, implications at paragraph ends, concession-then-redirect — might score the same or lower if it omits one surface marker.

**Fix:** Add structural checks: (a) example-before-generalization ordering within paragraphs, (b) subordinate clause ratio via spaCy dependency parsing, (c) "additive not sequential" transition pattern (the `ADDITIVE_TRANSITIONS` list is defined in the code but never used). Weight structural checks higher than surface marker checks — or report them as separate sub-scores so they can be analyzed independently.

### Flaw 3: The input schema conflates mode and sample

Each sample has a designated mode — either blog OR LinkedIn. This means a given article+notes pair is tested in only one mode. But the core research question asks: *does mode adaptation shift the tradeoff?*

To answer that question, you need the **same input tested in both modes**. Otherwise you cannot separate "this article+notes pair was harder" from "LinkedIn mode degrades substance." The study as designed compares different inputs across modes and attributes the difference to mode. That's a between-subjects design masquerading as a within-subjects analysis.

The Mann-Whitney U test proposed for mode comparison is technically correct for between-subjects — but the 2D tradeoff visualization implies within-subjects comparison, and the expected findings section interprets results as if mode is the independent variable. It isn't, because input varies simultaneously.

**Fix:** Run every input in both modes. This doubles the generation cost but is the only way to answer the mode question. If cost is a constraint, run a subset (15–20 samples) in both modes and the full set in primary mode only.

### Flaw 4: The pipeline condition conflates two uncontrolled variables

`qwen_haiku` combines Qwen generation with Haiku editing. If this condition scores poorly on substance, the study cannot determine whether Qwen lost the substance during generation or Haiku lost it during editing.

The README anticipates this: "That would suggest the Qwen generation step is already losing the notes before Haiku sees the output." But the study cannot confirm this without a fourth data point: the intermediate Qwen output, scored before Haiku edits it.

Without intermediate scoring, the study can observe the endpoint but not the mechanism. For an engineering study (Study 1), that's acceptable. For a research study about *where* fidelity degrades, it's a gap.

**Fix:** Save the intermediate Qwen output in the `qwen_haiku` pipeline. Score it on substance and voice. This enables: (a) measuring what Qwen loses from the notes, (b) measuring what Haiku recovers or damages, (c) attributing the pipeline outcome to a specific stage.

### Flaw 5: The NLI implementation for factual consistency is methodologically unsound

Two issues:

**Truncation bias.** The article premise is truncated to 1500 characters. Claims about content later in the article will score as "neutral" rather than "entailed." A model that accurately represents content from paragraph 10 would appear to be making unsupported claims. This systematically inflates contradiction rates for longer articles.

**Wrong model for the task.** `roberta-large-mnli` is trained on single-sentence NLI pairs (MNLI), not document-level factual consistency. This is a known weakness — dedicated factual consistency models exist specifically because NLI models perform poorly on this task at document scale. AlignScore, SummaC, or TRUE would be better choices.

**Fix:** Replace truncation with a retrieval step — for each output sentence, find the most relevant article passage (BM25 or embedding retrieval), then run NLI against that passage only. Or replace `roberta-large-mnli` with AlignScore, which is designed for this exact use case.

## Weaknesses (addressable, not fatal)

**1. N=30-50 per condition is thin for 6 condition-mode cells.** Wilcoxon signed-rank has low power at small N. With 30 paired samples, you need a large effect size (d > 0.7) to reach significance at α=0.05. If differences are moderate — which is more realistic for the substance metric — the study will produce inconclusive results and you won't know if it's a null finding or insufficient power.

**2. No baseline calibration of the substance metric.** What does cosine similarity of 0.5 mean? The threshold is arbitrary and unvalidated. You need: (a) a "ceiling" — Taya writes the blog post herself from the same notes, score that, (b) a "floor" — pure article summary with no notes, score that. This anchors the metric range and makes 0.5 either defensible or obviously wrong.

**3. No validation that the voice rubric correlates with human judgment.** The rubric produces a score, but there's no evidence it tracks "this sounds like Taya." Even one round of 10 samples scored by the rubric AND by Taya would establish whether the rubric measures what it claims. Without this, the rubric might be measuring "well-formatted prose" rather than "voice."

**4. `ADDITIVE_TRANSITIONS` is defined but never used.** Lines 28–31 of `voice_rubric.py` define patterns for "moreover," "however," "similarly" — signature markers in the taya-voice skill. No check function references them. This is a missing rubric item.

**5. `_paragraphs_end_on_consequence` is too permissive.** Matching `\bwhich\b` in the last sentence of a paragraph catches ordinary relative clauses that have nothing to do with drawing consequences ("a tool which is commonly used"). This will produce false positives constantly, inflating voice scores across all conditions uniformly and reducing the metric's discriminative power.

**6. No prompt documentation.** The study uses "the same strict style prompt" but doesn't specify or version-control it. If the prompt is worded differently for Qwen vs. Haiku (likely, given different model conventions), the comparison is partially contaminated by prompt quality rather than model capability.

**7. Temperature and sampling not controlled.** LLM outputs are stochastic. Running each condition once per sample means results include sampling variance. Even 2–3 runs per condition would allow estimating within-condition variance and separating model capability from sampling luck.

## Gaps

**1. No gold standard output.** The study has no ground truth for "good." Without even a small set of hand-written reference outputs (Taya writing the blog post herself from the same notes), all metrics are unanchored. Five hand-written references would calibrate the entire metric suite.

**2. No analysis of what substance is lost.** `substance_fidelity.py` logs per-claim scores but the study plan never analyzes which types of claims are dropped. Are opinions lost? Connecting logic? Specific details? If models consistently preserve factual claims but drop the user's interpretive framing, that's a specific and important finding — but the current analysis would report it as a single aggregate score.

**3. No cost analysis.** The study evaluates quality but not cost. Haiku API calls cost money; Qwen via HF free tier doesn't. If `haiku_only` outperforms `qwen_haiku`, the finding is incomplete without: how much more does it cost? If `qwen_haiku` matches `haiku_only`, the pipeline is strictly worse — same quality, higher cost, more complexity.

**4. No analysis of note complexity.** Notes vary wildly — bullet fragments vs. full arguments vs. scattered observations. The study doesn't control for or analyze how note structure affects results. A model might preserve simple bullet notes perfectly and lose complex argumentative notes. That would be the most practically useful finding, and it's invisible in the current design.

## V2 Proposal

### Changes to existing design

**Substance fidelity → Argument-level fidelity:**
- Decompose each note into typed components: `claim`, `evidence`, `logic` (connecting reasoning), `implication`
- Score each component type separately
- Report overall substance score AND component breakdown
- Flag "flattening" specifically: high claim preservation + low logic/implication preservation = model kept the topic, lost the thought

**Voice rubric → Two-tier scoring:**
- **Tier 1 (surface markers):** current rubric items — em-dash, "more specifically," contractions, banned words, no sequential transitions. Easy for models to mimic.
- **Tier 2 (structural patterns):** example-before-generalization ordering, subordinate clause ratio (spaCy dependency parse), additive-not-sequential transitions (use the existing `ADDITIVE_TRANSITIONS` list), concession-redirect presence. Hard for models to mimic.
- Report both tiers separately. The delta between them IS a finding: if Tier 1 is high and Tier 2 is low, the model is mimicking surface voice without capturing structural voice.

**Within-subjects mode comparison:**
- Run every input in both modes (blog + linkedin)
- Use paired tests (Wilcoxon signed-rank) across modes for the same input
- Total generations: 3 conditions × 2 modes × 30–50 inputs = 180–300 outputs

**Intermediate output capture:**
- For `qwen_haiku`, save Qwen's raw output before Haiku edits
- Score the intermediate on all metrics
- This creates a 4th implicit condition: `qwen_pre_edit`
- Enables delta analysis: what did Haiku add? what did Haiku damage?

**Fix NLI implementation:**
- Replace article truncation with BM25 retrieval: for each output sentence, retrieve the top-3 most relevant article passages, run NLI against those
- Or replace `roberta-large-mnli` with AlignScore

**Fix `_paragraphs_end_on_consequence`:**
- Remove `\bwhich\b` from consequence markers (too noisy)
- Add: `\bthe implication\b`, `\bthis is\b` (followed by consequence)
- Require the last sentence to contain BOTH a consequence marker AND a reference to the paragraph's main claim (coreference or lexical overlap)

### New additions

**Gold standard calibration set:**
- 5 samples where Taya writes the output herself from the same notes
- Score these through the full pipeline to establish metric ceilings
- 5 samples of pure article summary (no notes used) to establish metric floors
- Plot all model conditions relative to ceiling and floor

**Claim-type analysis:**
- Tag each note claim as: `opinion`, `fact`, `connection` (reasoning between claims), `question`
- Analyze which types survive transformation and which are dropped
- Hypothesis: connections and opinions are lost first; facts are preserved

**Cost-quality frontier:**
- Log estimated API cost per sample (token count × price)
- Plot substance × voice with cost as a third dimension (bubble size or color)
- Identify the efficient frontier: best quality per dollar

**Prompt versioning:**
- Store exact prompts in `prompts/{condition}_{mode}.txt`
- Include system prompt and user prompt separately
- Version-control these — prompt changes invalidate prior results

**Note complexity stratification:**
- Compute a complexity score per input: (number of claims) × (mean claim length) × (presence of connecting logic)
- Stratify results by complexity tercile (simple / moderate / complex)
- Report metrics separately per stratum

### What V2 makes possible that V1 cannot answer

| Question | V1 | V2 |
|---|---|---|
| Does the model preserve substance? | Yes (topic level only) | Yes (argument component level) |
| Which components are lost? | No | Yes — claim/evidence/logic/implication breakdown |
| Is the voice real or mimicked? | Cannot distinguish | Yes — Tier 1 vs Tier 2 delta |
| Where in the pipeline does loss occur? | No | Yes — intermediate output scoring |
| Does mode adaptation cause the loss? | No (between-subjects confound) | Yes (within-subjects, paired) |
| What is "good" on these metrics? | Unknown (no anchor) | Calibrated against gold standard and null baseline |
| Is this cost-effective? | Not measured | Yes — cost-quality frontier |
| Does note complexity predict failure? | Not measured | Yes — stratified analysis |
